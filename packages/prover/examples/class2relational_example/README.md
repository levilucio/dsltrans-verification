# Class2Relational Example: DSLTrans Transformation

This folder contains a **semantically meaningful** class diagram and its relational output using `class2relational_concrete.dslt`.

## Input model: document/catalog schema

The source model must be **well-formed for root containment**: one `Model` object with containment links `root` to each `Package`, and (if present) `rootAssociations` / `rootGeneralizations` to associations/generalizations. All example inputs and `gen_input.py` emit this structure.

The source model represents a **document/catalog** domain (one package, many entity types, attributes, and references):

| Concept | Count | Description |
|--------|-------|-------------|
| **Package** | 1 | `Pkg1` (root package) |
| **DataTypes** | 2 | `T1` (int-like), `T2` (string-like) |
| **Class C** (Category) | 50 | Entity types with `id` attribute typed by T1 |
| **Class D** (Document) | 80 | Entity types: 65 with `id` → T1, 15 with `ref` → C (FK to category) |
| **Class T1** (Tag1) | 40 | Entity types with `val` attribute → T2 |
| **Class T2** (Tag2/Item) | 40 | Entity types with `a` attribute → T2 |

- **Total: 210 classes**, 210 attributes, 423 objects, 632 links.
- All classes are concrete (`isAbstract=false`). Attribute names and classifier names use the spec’s finite vocabularies.

The input XMI is generated by `gen_input.py` (run from repo root: `python dsltrans_examples/class2relational_example/gen_input.py`). You can change the `n_c`, `n_d`, `n_t1`, `n_t2` counts there to scale the model.

## Small/medium example inputs (strict backward-link semantics)

For quick runs and tests under **strict backward-link semantics** (backward links must match existing traces; no create-fallback):

| File | Description | Expected output |
|------|-------------|-----------------|
| `minimal_input.xmi` | 1 package, 1 class C, 1 DataType, 1 attribute `a` | 1 Schema, 1 Table, 1 PK, ≥2 columns, `tables` link |
| `small_input.xmi` | 1 package, 2 classes C/D, 2 DataTypes, C.id→T1, D.ref→C, D.val→T2 | 1 Schema, 2 Tables, ≥1 FK (ref), `tables` links |
| `medium_input.xmi` | 1 package, 3 classes C/T1/T2, 2 DataTypes, one attribute per class | 1 Schema, 3 Tables (C, T1, T2), `tables` links |
| `with_association_input.xmi` | 1 package, 2 classes C/D, 1 Association with 2 AssociationEnds, 1 AssocBetween (1:1 C–D) | 1 Schema, ≥2 Tables, ≥1 FK (1:1 rule), traces |
| `with_generalization_input.xmi` | 1 package, 2 classes (Parent abstract, Child concrete), 1 Generalization | 1 Schema, ≥1 Table, inheritance/discriminator handling, `tables` link |
| `with_many_to_many_input.xmi` | 1 package, 2 classes C/D, 1 Association (A2), 1 AssocBetween (N:M, srcUpper/tgtUpper=16) | 1 Schema, ≥3 Tables (C, D, junction), `tables` link |

The test `test_class2relational_example_inputs_strict_semantics` in `tests/test_dsltrans_runtime_engine.py` runs the transformation on minimal/small/medium and the association/generalization/many-to-many inputs and asserts the expected structure. The test `test_class2relational_all_example_inputs_smoke` runs every `*_input.xmi` in this folder and checks that the output has at least one Schema, one Table, and a `tables` link.

- **Output:** 1536 nodes, 1180 edges, 841 traces.
- **Concrete properties:** See status semantics below.

## Property status semantics

| Status | Meaning |
|--------|---------|
| `holds` | Every precondition match has a valid postcondition match (with traces); property satisfied |
| `violated` | Normal property: at least one precondition match has no valid postcondition |
| `violated` | ShouldFail property: no precondition match had a postcondition (correct; transformation is OK) |
| `unexpectedly_holds` | ShouldFail property: at least one pre-match had a post-match; transformation may have produced incorrect output |
| `precondition_never_matched` | Precondition pattern never found in input model; property cannot be evaluated |
## How to run

From the `symbolic-execution-engine` root:

```bash
# Optional: regenerate input (e.g. after editing gen_input.py)
python dsltrans_examples/class2relational_example/gen_input.py

# Run transformation and write Option B artifacts + property results
python -m symex_engine.dsltrans.run \
  --spec dsltrans_examples/class2relational_concrete.dslt \
  --in dsltrans_examples/class2relational_example/classdiagram_input.xmi \
  --out dsltrans_examples/class2relational_example/relational_output.xmi \
  --transformation Class2Relational_frag_canonical \
  --artifact-dir dsltrans_examples/class2relational_example \
  --check-concrete-properties

# Use a custom property report path
python -m symex_engine.dsltrans.run ... --check-concrete-properties --property-report my_report.json
```

Optionally add `--source-ecore` and `--target-ecore` for metamodel consistency checks.
